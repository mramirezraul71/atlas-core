Second-wave patch: integrate market_open_config.json into risk runtime

Integrate market_open_config.json as the operational source of truth for risk
and schedule parameters consumed via config.settings.TradingConfig.

Summary:
- Central loader _apply_market_open_operational_config reads
  atlas_code_quant/config/market_open_config.json once per TradingConfig init
  (and refresh_market_open_operational_config() can re-read the file without
  touching other subsystems).
- From JSON risk (when the corresponding env is not explicit): max_positions
  -> market_open_max_positions; kelly_fraction -> kelly_fraction;
  max_risk_per_trade -> equity_kelly_max_risk_per_trade_pct;
  max_position_pct -> kelly_max_position_pct; daily_loss_limit ->
  market_open_daily_loss_limit_pct.
- From JSON schedule_et (when env does not override): market_open,
  market_close, scan_interval_min -> market_open_scan_interval_min alongside
  existing schedule fields used by the market hours gate.
- Helpers validate HH:MM and numeric ranges; _finalize_market_open_operational_clamps
  applies final safe caps.
- _sync_risk_fields_from_explicit_env re-applies explicit env at the start of
  each load so precedence env > JSON > defaults is effective even when process
  env is set before TradingConfig is instantiated (dataclass defaults for
  _fenv/_ienv are fixed at import).

Precedence (per field):
- Explicit non-empty process environment variable > market_open_config.json >
  code/dataclass defaults.
- Documented in market_open_config.json under _precedence and in the method
  docstring.

Error handling:
- Missing/unreadable JSON or non-object root: logger.warning, no crash on boot,
  market_open_config_loaded=False, keep env/defaults.
- Single invalid field: logger.warning for that field, keep the field's
  already-resolved value, apply the rest of the file.
